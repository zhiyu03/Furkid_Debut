# 毛孩子出道计划 — 移动端重构执行方案

> 目标：以**手机屏幕尺寸**为主布局，支持头饰 / 妆容 / 造型 / 衣物多选，汇总后调用 **Replicate `openai/gpt-image-2`** 完成一张「已变装」合成图。

---

## 0. 安全与密钥（必须先做）

- **切勿**把 `REPLICATE_API_TOKEN` 写入前端代码或提交到 Git；仅放在服务端 `.env`。
- 若密钥曾在聊天、截图或仓库中泄露，请到 [Replicate Account / API tokens](https://replicate.com/account/api-tokens) **立即撤销并重新生成**。
- 客户端只调用**自家后端**（如 `POST /api/debut`），由后端读取 `process.env.REPLICATE_API_TOKEN` 调用 Replicate。

---

## 1. 需求摘要（对照你的产品描述）

| 序号 | 需求 | 实现要点 |
|------|------|----------|
| 1 | 用户上传宠物图 | 保留 `multipart/form-data` 或 `base64`；移动端压缩大图（可选，见第 6 节） |
| 2 | 主区域布局：宠物图约占屏宽 1/3，右侧为四类入口 | 使用 **flex 横向**：左列 `~33vw`（或 `flex: 0 0 33%`）放主图；右列 `~67%` 放「头饰 / 妆容 / 造型 / 衣物」四个入口 |
| 3 | 点某类进入子面板，展示该类下的多个**图标**；点选一个图标后，该图标出现在**宠物图下方** | 右列二级：抽屉 / 底部 Sheet / 全屏覆盖层展示图标网格；选中项写入 `selectedItems[]`，主图下方为「已选槽位」横向滚动条 |
| 4 | 可多类、多件叠加选择 | 状态结构支持多 `categoryId + itemId`；同一类可允许多件或限制 1 件（产品规则见第 4 节） |
| 5 | 「开始变装」→ 调用 image-2 合成一张结果图 | 后端组装 `prompt` + `input_images`（见第 5 节），`replicate.run("openai/gpt-image-2", { input })` |

## 2. 信息架构与页面状态

建议用**单页多状态**（与现有 `App.jsx` 思路一致），拆成清晰子组件：

```
MobileShell（max-width: 430px 居中 + min-h-dvh）
├── Header（标题 / 可选返回）
├── MainRow（横向主区）
│   ├── PetColumn（宽 ~33%）
│   │   ├── PetImage（上传后展示；未上传为占位 + 上传按钮）
│   │   └── SelectedStrip（已选图标条，宠物图「下方」）
│   └── CategoryRail（宽 ~67%）
│       ├── 4 个大入口按钮：头饰 / 妆容 / 造型 / 衣物
│       └── （展开时）CategoryPanel：当前类的 item 图标网格
├── FooterCTA：「开始变装」（校验：已上传图 + 至少 N 项选择，规则可配）
└── ResultOverlay 或独立 Result 步：对比原图 / 结果 + 重选 / 再生成
```

**交互细节建议**

- 小屏下若 1/3 + 2/3 过挤：可改为「上图下栏」但**视觉权重**仍为「图区高度约 1/3 屏」：例如 `pet` 区域 `min-height: 33vh`，下方 `CategoryRail` 占剩余空间；你原文强调「右侧」则以横排为默认，**横屏**或大屏自动更舒服。
- 「图标在宠物图下方」：实现为 `PetColumn` 内纵向：`Image` + `SelectedStrip`（横向 `scroll-x`）。

## 3. 前端重构任务清单

| 阶段 | 任务 | 产出 |
|------|------|------|
| F1 | 建立移动端视口约束：`viewport` meta、`max-w-[430px]`、`mx-auto`、`safe-area-inset` | 全局布局与 Douyin WebView 兼容基线 |
| F2 | 拆分组件：`MobileShell`、`PetColumn`、`CategoryRail`、`CategoryPanel`、`SelectedStrip`、`DebutButton`、`ResultView` | 可维护目录结构 |
| F3 | 状态模型：`uploadedFile` / `previewUrl`；`activeCategory`；`selected: { category, item }[]` 或按类 `Record<CategoryId, ItemId[]>` | 单一数据源，便于拼 prompt |
| F4 | 资源：`/public/items/{category}/{id}.png` 或 SVG 清单 JSON（id、label、promptFragment） | 设计可替换素材管线 |
| F5 | 上传：复用/精简 `ImageUpload`（仅保留相册 + 可选相机），限制类型与大小 | 与后端一致 |
| F6 | 调用：「开始变装」`FormData`：`image` + `selections`（JSON）或只传 `itemIds` 由后端映射 prompt | 不暴露 Token |

**Tailwind 建议**

- 使用 `h-[33vh]` 或 `flex-[0_0_33%]` 等与产品一致的占位；右栏 `overflow-y-auto` 防止面板过长溢出。

## 4. 业务规则（需你拍板，写进 PRD 常量）

在代码里用常量表维护，避免散落魔法字符串：

- **每类最多可选几件？**（例如头饰 2、衣物 1…）
- **同类多选时 prompt 如何拼接？**（逗号列表 / 优先级只取最后一件）
- **未选任何饰品是否允许生成？**（建议：允许，但 prompt 仅做「精修宠物照」类弱指令）

## 5. 后端与 Replicate 集成方案

### 5.1 依赖

```bash
cd server && npm install replicate
```

### 5.2 环境变量

| 变量 | 说明 |
|------|------|
| `REPLICATE_API_TOKEN` | 服务端专用，勿下发浏览器 |

### 5.3 `gpt-image-2` 输入约定（对齐官方示例）

- `input_images`：**需可访问的图片 URL 数组**。用户上传文件后，有两种常见做法：
  1. **服务端临时托管**：上传保存到 `uploads/`，对外通过 `https://你的域名/uploads/xxx` 访问（生产需 HTTPS）；开发可用 ngrok 暴露公网 URL。
  2. **Replicate 文件上传**：若后续改用官方 file API，把本地文件上传得到 URL 再传入（可列为二期）。

当前黑客松路径：在 **同一次请求** 内 `multer` 存盘 → 构造**绝对可访问**的 `imageUrl`（注意手机访问开发机时需局域网 IP + 端口，或部署后域名）。

### 5.4 Prompt 组装策略

- 为每个 `itemId` 配置短英文 **片段**（如 `red wool sweater`, `golden crown`）。
- 总 `prompt` 模板示例结构：

  ```
  Keep the same pet identity and pose from the input photo. Apply the following styling as a cohesive outfit:
  - Headwear: ...
  - Makeup: ...
  - Hairstyle / fur styling: ...
  - Clothing: ...
  Photorealistic, soft studio lighting, mobile-friendly vertical composition.
  ```

- `aspect_ratio`：**Replicate `gpt-image-2` 仅允许** `"1:1"`、`"3:2"`、`"2:3"`；竖屏用 **`2:3`**（勿再传 `9:16` 等，否则会 422）。

### 5.5 调用与落盘（Node ESM 示例骨架）

逻辑要点（**不要**把 Token 写进代码）：

- `import Replicate from "replicate"`
- `const replicate = new Replicate({ auth: process.env.REPLICATE_API_TOKEN })`
- `const output = await replicate.run("openai/gpt-image-2", { input })`
- 与官方示例一致：`output[0].url()` 取 URL；用 `fetch` + `writeFile` 或 `Readable` 管道写入 `server/uploads/debut-{uuid}.webp`，返回 `/uploads/...` 给前端展示。

### 5.6 新 API 路由建议

| 方法 | 路径 | 作用 |
|------|------|------|
| `POST` | `/api/debut` | 接收图片 + 选项 JSON → 拼 prompt → Replicate → 保存结果 → `{ resultUrl, debugPrompt? }` |

旧路由 `/api/generate`、`/api/score`：**标记 deprecated**，或删除并在 `README` / `CLAUDE.md` 中更新，避免双套逻辑。

### 5.7 错误与超时

- Replicate 排队与生成较慢：前端按钮 loading + 可取消（可选）；后端 `fetch` 设置合理超时或异步 webhook（二期）。
- 统一返回 `{ error: string, code? }`，前端 Toast 替代 `alert`。

## 6. 非功能需求

- **体积**：移动端上传前可用 `canvas` 压缩 JPEG（例如长边 1024），减轻带宽与 API 成本。
- **无障碍**：图标按钮加 `aria-label`。
- **可测性**：`promptBuilder` 与选项映射**纯函数**单测，避免回归。

## 8. 验收标准（Checklist）

- [ ] 在 iPhone Safari / Chrome 设备模式宽度下，主图 + 右侧四入口无横向错位溢出。
- [ ] 多类多选后，已选图标均显示在宠物图下方条带，可删除单项。
- [ ] 点击「开始变装」后能看到**一张**合成结果图（非 mock）。
- [ ] 仓库内无 Token；`.env.example` 仅含变量名说明。
- [ ] 失败场景有用户可读提示（网络、Replicate 错误、未上传图）。

## 9. 风险与依赖

- **`input_images` 必须公网可达**：本地开发若仅用 `localhost`，Replicate 无法拉取；需部署或使用隧道/局域网可被外网访问的 URL（团队需统一一种方案）。
- **模型与计费**：`gpt-image-2` 按次计费，需在控制台监控用量。
- **合规**：用户上传宠物照涉及隐私，README 中声明「不上传第三方除推理服务外存储」等（按你们比赛要求调整）。
