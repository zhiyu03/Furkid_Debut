# 出道打分与角色系统 — 阶段 1 执行方案

> **目标**：在用户完成装扮选择并生成定妆图后，根据 **catalog 选择项** 计算 **0～100 出道潜力分**，映射 **档位**，并输出 **丰富的出道角色**（如女明星、C 位、练习生等）；与既有 **「毛孩子出道计划」叙事**、**结果页 UI**、**一键复制分享文案** 打通。  
> **阶段 1 原则**：规则打分、可解释、零额外模型费用；**不**声称对生成图像做计算机视觉识妆（该能力列为阶段 2 可选）。

---

## 1. 背景与对齐

| 已有能力 | 本阶段衔接方式 |
|----------|----------------|
| `catalog/itemCatalog.json` 四品类：头饰 / 妆容 / 造型 / 衣物 | 作为打分唯一数据源 |
| `POST /api/debut` 接收 `selections` | 服务端用同一数组计算分数与角色，写入 JSON 响应 |
| `client` 结果页 + 复制文案 | 展示分数、档位、角色卡；分享串模板带入 `{{score}}` `{{roleTitle}}` `{{tagline}}` |
| `server/services/debutPrompt.js` | **不改** prompt 职责边界；打分独立模块 |

---

## 2. 阶段范围与非目标

**范围内**

- 基于 `selections: { categoryId, itemId }[]` 的规则引擎。
- 档位 + 每档多角色池 + 稳定哈希选角（同选择同分 → 同角色，微调选择 → 易换角）。
- API 响应扩展；前端结果区 UI + 分享文案动态化。

**范围外（本执行单不实现）**

- 对结果图调用 Vision API 判妆。
- 水印 / 海报导出图（可另立文档）。
- 排行榜、服务端持久化用户分数。

---

## 3. 评分规则（建议定稿数值，实现时写入 `debutScore.js`）

### 3.1 基础分：按选中条目累加

每个被选中的 `itemId` 对应一条 **基础分**（可配置表，与 catalog 同步维护）。**建议初值**（总和易落在 40～90，再由加成与封顶拉满区间）：

| 品类 | 说明 | 建议每条基础分 |
|------|------|------------------|
| `makeup` 妆容 | 阶段 1 叙事重点「妆容」 | **12** |
| `headwear` 头饰 | | 10 |
| `styling` 造型 | | 10 |
| `clothing` 衣物 | max 1 | 14 |

> 若某 `itemId` 未在分表内，回退默认分 **8**，并打 `console.warn` 便于补表。

### 3.2 完成度加成

- 四个品类 **均至少 1 项选择**：**+10**「全妆出道预备」。
- 仅缺一类：不加分或 **+3**（二选一，产品定）。

### 3.3 妆容侧重（与「根据妆容判断」话术一致）

- `makeup` 选中 **≥2** 项：**+5**。
- `makeup` 选中 **≥3** 项（顶格）：再 **+3**（与 `maxPerCategory.makeup=3` 对齐）。

### 3.4 封顶与取整

- 原始分和 **上限 100**，下限 **0**，输出 **整数** `score`。
- 可选：对「完全未选」给保底 **15** 并档位锁定为最低档（避免 0 分冷场）— 产品若要求「必须选满才能生成」则当前流程已要求有图，仍可能 `selections=[]`，需定义：**空选择**是否允许生成（现有 prompt 允许）；若允许，建议保底分 + 最低档角色 **「素颜围观群众」** 类。

### 3.5 档位 `tierId`（与角色池绑定）

| `tierId` | 分数区间 | 对用户的档位名（示例） |
|----------|----------|------------------------|
| `trainee` | 0–34 | 练习生 |
| `backup` | 35–54 | 预备役 / 伴舞位 |
| `rising` | 55–69 | 上升期新人 |
| `center` | 70–84 | 准 C 位 |
| `diva` | 85–100 | 本番出道 / 女明星档 |

文案上可称 **「出道潜力指数」**，脚注：**娱乐向规则评分，非专业评审**。

## 4. 角色池设计（丰富度）

### 4.1 数据结构（新建 `catalog/debutRoles.json` 或 `server/data/debutRoles.json`）

建议 Schema：

```json
{
  "tiers": [
    {
      "id": "trainee",
      "label": "练习生",
      "roles": [
        { "title": "剧组场务喵", "tagline": "离镜头还有三条小鱼干", "emoji": "🎬" },
        { "title": "旁听练习生", "tagline": "C 位在隔壁，你先蹲个机位", "emoji": "🪑" }
      ]
    }
  ]
}
```

每档 **`roles.length ≥ 6`**，优先覆盖：练习生、伴舞、综艺咖、女团风、男团风、**C 位**、**女明星**、**限定团宠**、**夜场主理**、**国风出道**、**街头爱豆**、**反差萌** 等差异维度。

### 4.2 选角算法（伪代码）

```
inputs: selections[], score
tier = tierFromScore(score)
roles = rolesByTier[tier]
key = stableStringify(selections sorted by categoryId, itemId)
hash = simpleHash(key) // 32-bit int
index = hash % roles.length
role = roles[index]
```

返回对象示例：

```json
{
  "score": 78,
  "tierId": "center",
  "tierLabel": "准 C 位",
  "role": {
    "title": "限定团宠 ACE",
    "tagline": "镜头感已点满，就差一条热搜",
    "emoji": "⭐"
  }
}
```

## 5. API 契约变更

### 5.1 `POST /api/debut` 成功响应（`mock: true` / `false` 均一致带上）

在现有字段基础上增加：

| 字段 | 类型 | 说明 |
|------|------|------|
| `debutScore` | `number` | 0–100 |
| `debutTierId` | `string` | 如 `center` |
| `debutTierLabel` | `string` | 展示用中文档名 |
| `debutRole` | `{ title, tagline, emoji }` | 当前选中角色 |

**计算时机**：在 `buildDebutPrompt(selections)` 前后均可；必须在返回 JSON 前完成。

### 5.2 错误响应

不变；失败时不返回打分。

### 5.3 错误文案统一（顺带）

将 `500` 中 `变装失败` 等与 **出道定妆** 叙事一致的文案列入同一 MR（若尚未改）。

## 7. 服务端模块划分

| 文件 | 职责 |
|------|------|
| `server/services/debutScore.js` | `scoreSelections(selections) → { score, tierId, tierLabel }`；读 catalog 校验 id。 |
| `server/services/debutRolePick.js` | `pickRole(tierId, selections) → role`；读 `debutRoles.json`。 |
| `server/routes/debut.js` | 组装并 `res.json({ ..., debutScore, ... })`。 |

单元测试（可选但推荐）：对几组固定 `selections` 断言分数区间与 `tierId`。

## 8. 实施顺序（建议工单拆分）

1. **数据**：新增 `debutRoles.json`，每档 ≥6 角色；`itemScore` 表（可与 `debutScore.js` 内联常量首版）。

## 9. 验收标准（DoD）

- [ ] 任意成功生成均带 `debutScore`、`debutRole`，且分数在 0～100。  
- [ ] 改任意一项选择，分数或角色至少一类有变化概率高（哈希选角在池内轮换）。  
- [ ] 分享复制内容含分数 + 角色标题 + 至少一个话题标签。  
- [ ] 结果页有 **娱乐向非专业评审** 提示。  
- [ ] Mock 与真实 Replicate 路径行为一致。
